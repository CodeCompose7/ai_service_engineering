## 2. 무엇을 재나 — Hit Rate@k와 MRR

질문마다 검색 결과에서 정답 근거가 몇 위에 나오는지를 보고, 그 순위로 두 지표를 계산합니다.

```mermaid
flowchart TB
  Q["질문"] --> S["top-k 검색 결과"]
  S --> R["정답 근거가 처음 나온 순위"]
  R --> RC["Hit Rate@k<br/>k 안에 있으면 1"]
  R --> MR["MRR<br/>1 / 순위"]
  classDef default rx:8,ry:8;
```

### 2.1. Hit Rate@k — 상위 k 안에 정답이 있나

여기서 `k`는 검색 결과 중 위에서 몇 개까지 볼지를 정하는 자릅니다. mini RAG는 상위 k개 청크만 LLM에 넘기므로, 정답 근거가 그 k개 안에 들었는지가 중요합니다.

한 질문의 Hit Rate@k는 상위 k개 안에 정답 근거가 있으면 1, 없으면 0입니다. 1위든 k위든 안에만 들면 똑같이 1이라, 위치는 보지 않고 "빠뜨리지 않고 데려왔나"만 봅니다.

```mermaid
flowchart TB
  subgraph L["검색 결과 (유사도 순)"]
    direction TB
    A["1위 · 관련 없음"] --> B["2위 · 관련 없음"] --> C["3위 · 정답 근거 ★"] --> D["4위"] --> E["5위"]
  end
  classDef default rx:8,ry:8;
  classDef hit fill:#eaf2ff,stroke:#4a78c0;
  class C hit;
```

정답이 3위에 있으면 k에 따라 점수가 달라집니다. Hit Rate@1과 Hit Rate@2는 0이고, 3위가 잡히는 Hit Rate@3부터 1입니다. k를 키울수록 정답을 담기 쉽지만, 관련 없는 청크도 그만큼 함께 딸려 옵니다.

여러 질문의 이 0과 1을 평균낸 것이 적중률입니다. 5문항 중 4문항에서 정답이 상위 k에 들면 0.80입니다.

정확한 이름은 Hit Rate@k(또는 Success@k)입니다. 검색 평가의 표준 짝인 precision/recall과는 이렇게 다릅니다.

| 지표 | 식 | 묻는 것 |
| --- | --- | --- |
| Precision@k | top-k 중 관련 수 / k | 가져온 k개 중 몇 개가 정답인가 |
| Recall@k | top-k 중 관련 수 / 전체 관련 수 | 정답 중 몇 개를 가져왔나 |
| Hit Rate@k | top-k에 관련이 하나라도 있으면 1 | 정답을 빠뜨리지 않았나 |

우리 평가셋은 질문마다 정답 근거가 하나라 전체 관련 수가 1입니다. 그때 Recall@k는 0/1 또는 1/1이 되어 Hit Rate@k와 같아지므로, 여기서는 둘이 같은 값입니다. 정답이 여럿이면 Recall@k는 비율이 되고, Precision@k는 가져온 k개의 정확도를 봅니다. RAG는 top-k를 통째로 LLM에 주므로, 정답을 담았는지를 보는 이 지표와 순위를 보는 MRR을 함께 씁니다.

### 2.2. MRR — 얼마나 위에서 찾았나

적중률이 "들었나"만 본다면, MRR은 "몇 위로 들었나"까지 봅니다. 한 질문의 점수는 정답 근거가 처음 나온 순위의 역수, 즉 1/순위입니다.

```mermaid
flowchart LR
  P1["1위<br/>1/1 = 1.00"] -.- P2["2위<br/>1/2 = 0.50"] -.- P3["3위<br/>1/3 = 0.33"] -.- P4["4위<br/>1/4 = 0.25"] -.- P5["5위<br/>1/5 = 0.20"]
  classDef default rx:8,ry:8;
```

역수라서 위쪽일수록 점수가 가파르게 큽니다. 1위에서 2위로 내려가면 1.0에서 0.5로 절반이 깎이지만, 4위에서 5위로는 0.25에서 0.2로 조금만 줄어듭니다. 정답을 맨 위로 올리는 일이 그만큼 값지다는 뜻입니다. 상위 k 안에 정답이 아예 없으면 0입니다.

이 1/순위를 여러 질문에 평균낸 것이 MRR입니다. 같은 적중이라도 정답을 1위로 올린 검색이 더 높은 MRR을 받습니다.

두 지표는 보는 각도가 다릅니다. Hit Rate@k는 k 안 어디든 들면 같은 점수라 순위의 질을 못 보고, MRR은 같은 적중이라도 1위로 올린 쪽에 더 높은 점수를 줍니다. 세 질문의 정답 순위가 이렇다고 해 봅니다.

| 질문 | 정답 순위 | Hit Rate@5 | 1/순위 |
| --- | --- | --- | --- |
| Q1 | 1위 | 1 | 1.00 |
| Q2 | 4위 | 1 | 0.25 |
| Q3 | 5위 밖 | 0 | 0.00 |
| 평균 | | 0.67 | 0.42 |

Hit Rate@5는 셋 중 둘을 찾아 0.67, MRR은 순위까지 반영해 0.42입니다. 둘을 함께 보면 정답을 빠뜨리지 않으면서 위로 올리는지 한눈에 드러납니다.

## 3. 평가셋 — 질문과 정답 근거

평가에는 정답이 정해진 질문 묶음이 필요합니다. 질문마다 정답 근거로 쓸 문구를 정해 두고, 검색된 청크 텍스트에 그 문구가 있으면 정답으로 칩니다.

```python
EVAL_SET = [
    {"q": "검색 증강 생성은 무엇인가요?", "answer": "새로운 정보를 검색"},
    {"q": "참조 데이터는 어디에 저장되나요?", "answer": "벡터 데이터베이스"},
    {"q": "검색된 정보는 어떻게 LLM에 전달되나요?", "answer": "프롬프트 엔지니어링"},
    ...
]
```

청크 경계는 설정마다 달라지지만 정답 문구는 그대로라, 어떤 청킹 설정이든 같은 잣대로 비교됩니다. PDF 추출의 군더더기 공백을 피하려고 공백을 지우고 맞춥니다.

## 4. 청킹 조합 비교

청크 크기와 overlap을 바꿔가며 같은 평가셋으로 점수를 냅니다. 인덱스를 설정마다 새로 만들어(다시 청킹·임베딩) 검색 품질을 잽니다.

크기는 완전성과 정밀도의 맞바꿈입니다. 너무 작으면 답이 갈려 정답을 놓쳐 적중률이 떨어지고, 너무 크면 한 청크에 여러 주제가 섞여 정답이 위로 잘 안 올라와 MRR이 떨어집니다.

```mermaid
  MAIN --> EC["evaluate_config<br/>인덱싱 → 검색 → 채점"]

```

```bash
uv run python src/section2/lec07/evaluate.py
```

```text
평가셋 5문항, top-5 검색

 size overlap chunks     Hit@5   MRR
  300      50     66      0.80  0.80
  500      80     39      1.00  1.00
 1000     150     19      1.00  0.87
```

읽어낼 점입니다.

- 청크가 작은 300은 Hit Rate@5가 0.80으로, 다섯 문항 중 하나를 놓칩니다. 답이 작은 청크로 갈려 상위 5개 밖으로 밀린 것입니다.
- 500은 적중률과 MRR이 모두 1.00입니다. 다섯 문항 모두 1위로 찾았습니다. lec03부터 써 온 기본 설정이 이 데이터에서 가장 좋습니다.
- 1000은 다 찾지만(적중률 1.00) MRR이 0.87로 떨어집니다. 청크가 커 여러 주제가 섞이니 정답이 1위가 아닌 경우가 생깁니다.
- 정답이 한 설정에 고정돼 있지 않습니다. 데이터와 질문에 따라 다르므로, 이렇게 숫자로 재서 고릅니다.

## 6. 웹에서 조회·평가하기

같은 기능을 브라우저에서도 쓸 수 있게 작은 FastAPI 앱을 둡니다. [app.py](../../../src/section2/lec07/app.py)는 질문을 검색·생성하는 조회와 EVAL_SET 평가를 웹으로 제공합니다.

```bash
uv run uvicorn section2.lec07.app:app
# 브라우저에서 http://127.0.0.1:8000 을 연다
```

```mermaid
flowchart LR
  B["브라우저"] --> Q["/api/query<br/>검색·생성 + 근거"]
  B --> E["/api/eval<br/>EVAL_SET 채점"]
  Q --> RAG["lec06 mini RAG"]
  E --> EV["lec07 evaluate"]
  RAG --> IDX["lec05 벡터DB"]
  EV --> IDX
  classDef default rx:8,ry:8;
```

- 질문 조회: 질문을 넣으면 mini RAG가 답과 근거(청크 순위·유사도)를 보여줍니다.
- EVAL_SET 평가: 버튼 한 번으로 평가셋 문항별 적중 순위와 Hit Rate@k·MRR을 표로 봅니다.

임베딩 모델은 서버를 띄운 뒤 첫 요청에서 한 번만 메모리에 올라가고, 그 뒤 요청은 빠릅니다. lec05에서 말한 "한 번 로드, 여러 번 서빙"을 실제 서버로 보이는 셈입니다. 단발 CLI가 매번 모델을 다시 올리던 것과 달리, 서버는 한 번만 올립니다. 생성은 lec06과 같이 LiteLLM을 거치므로 `DEFAULT_PROVIDER`만 바꾸면 클라우드·로컬 어느 모델로도 같은 앱이 돕니다.

## 7. 정리

- 검색 평가는 정답이 정해진 질문 묶음으로 검색이 근거를 잘 가져오는지 재는 일입니다.
- Hit Rate@k는 정답을 상위 k 안에 넣었는지, MRR은 얼마나 위에서 찾았는지를 봅니다.
- 청크 크기는 완전성과 정밀도의 맞바꿈이고, 좋은 설정은 데이터마다 달라 측정으로 고릅니다. 여기서는 500이 가장 좋았습니다.
- 이 잣대가 있으면 청킹·overlap·임베딩·검색 개수를 바꿀 때마다 좋아졌는지 나빠졌는지를 숫자로 확인할 수 있습니다. 이것으로 S2의 mini RAG 한 바퀴와 그 검색을 재는 잣대가 손에 들어옵니다.
